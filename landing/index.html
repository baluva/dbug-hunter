Ce que DBug Hunter trouve

14 contrôles, 4 familles de bugs

Chaque problème est remonté avec sa table, sa colonne, un exemple et une gravité.

🔗

Intégrité référentielle

  • • Clés étrangères orphelines (parent inexistant)
  • • Lignes strictement dupliquées
  • • Doublons sur colonnes uniques (email, sku…)
🧪

Qualité des données

  • • Types de stockage mélangés dans une colonne
  • • E-mails invalides · dates invalides ou futures
  • • Valeurs négatives suspectes · NULL manquants
  • • Espaces superflus, chaînes vides, outliers (IQR)
🧱

Schéma

  • • Table sans clé primaire
  • • Table vide
⚡

Performance

  • • Clé étrangère sans index (jointures lentes)

Pour les développeurs

Le moteur, sans l'interface

La détection est découplée de l'UI : utilisez-la en bibliothèque Python ou via une API REST FastAPI (doc Swagger auto-générée sur /docs).

  • ✓ 1 fonction = 1 contrôle : extensible en quelques lignes
  • ✓ Couvert par des tests pytest
  • ✓ Zéro dépendance front, déployable partout
python 
from dbughunter import scan_database

report = scan_database("ma_base.sqlite")
print(report["summary"]["score"])  # 0–100
for f in report["findings"]:
    print(f["severity"], f["table"], f["title"])

Questions fréquentes

Quels formats de base sont acceptés ?+

Tout fichier SQLite : .db, .sqlite, .sqlite3, .db3.

Mes données sont-elles conservées ?+

Non. Le fichier est analysé en mémoire puis supprimé : rien n'est stocké côté serveur.

Comment est calculé le score /100 ?+

Une base sans problème part de 100 ; chaque finding retire des points selon sa gravité (critique > avertissement > info).

Puis-je l'utiliser hors interface web ?+

Oui — en bibliothèque Python (scan_database()) ou via l'API REST FastAPI.
